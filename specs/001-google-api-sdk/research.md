# Research: Google API Python SDK

**Phase 0 Output** | **Date**: 2026-02-23

## R1: HTTP Client Choice

- **Decision**: httpx
- **Rationale**: Natively provides both `Client` (sync) and `AsyncClient` (async) with identical APIs. Used by OpenAI, Anthropic, and Cloudflare SDKs (all Stainless-generated). Supports HTTP/2.
- **Alternatives considered**: aiohttp (~50% faster for high-concurrency but async-only, would require maintaining two HTTP backends); requests (sync-only, no HTTP/2)

## R2: Sync/Async Strategy

- **Decision**: Async-first source + unasync build-time generation
- **Rationale**: Single codebase maintaining both sync and async variants. Proven by httpcore, elasticsearch-py, urllib3. Eliminates behavioral drift between sync/async.
- **Alternatives considered**: Manual dual implementations (maintenance burden, subtle bugs); sync-only (insufficient for modern async workloads)

## R3: Response Models

- **Decision**: Pydantic v2
- **Rationale**: Industry standard for Python API SDKs. Provides validation, serialization, IDE support via type hints, and `alias` for camelCase->snake_case mapping. `extra="allow"` handles new Google API fields gracefully.
- **Alternatives considered**: msgspec (~5x faster JSON decoding but lacks ecosystem); dataclasses (no validation/serialization); attrs (less ecosystem support for API SDKs)

## R4: Authentication Foundation

- **Decision**: Build on `google-auth` + `google-auth-oauthlib`, add pluggable `TokenStore`
- **Rationale**: Official Google libraries handle credential types, token refresh, and JWT signing. Adding TokenStore fills the explicit gap in credential storage that no existing library covers.
- **Alternatives considered**: Custom OAuth implementation (massive maintenance burden, as aiogoogle discovered); pure google-auth without storage abstraction (poor DX for desktop apps)

## R5: Token Storage Backends

- **Decision**: Three built-in backends: FileTokenStore (default), KeyringTokenStore, EnvTokenStore
- **Rationale**: Covers the three primary deployment scenarios: development (file), desktop apps (OS keychain), CI/CD (env vars). Custom backends via ABC for cloud secret managers.
- **Alternatives considered**: Single file-based only (insufficient for production); database backend (over-engineering for v1)

## R6: Pagination Pattern

- **Decision**: Auto-paginating async iterators yielding Pydantic models
- **Rationale**: Matches OpenAI/Anthropic SDK pattern. Users never touch pageTokens. Sync variant generated by unasync uses `__iter__`/`__next__`.
- **Alternatives considered**: Manual page token passing (poor DX); loading all results into memory (unbounded memory for large result sets)

## R7: Retry Strategy

- **Decision**: Transport-level middleware with exponential backoff + jitter
- **Rationale**: Google recommends `wait = min(2^n + random_ms, max_backoff)` retrying on 429, 500, 502, 503, 504. Transport-level keeps service code clean.
- **Alternatives considered**: Per-method retry decorators (code duplication); tenacity library (unnecessary dependency for simple retry logic)

## R8: Upload Strategy

- **Decision**: Auto-select simple upload (<5 MB) vs resumable upload (>=5 MB)
- **Rationale**: Resumable uploads add ~10x overhead for small files. 5 MB threshold matches Google's recommendation. Resumable uploads use 8 MiB chunks (must be multiples of 256 KiB).
- **Alternatives considered**: Always resumable (poor performance for small files); always simple (fails for large files, no progress tracking)

## R9: Testing Strategy

- **Decision**: respx (unit) + VCR.py (integration) + optional e2e
- **Rationale**: respx mocks httpx at transport level for fast isolated unit tests. VCR.py records real API interactions as cassettes for integration tests that catch response shape changes. E2e runs on schedule in CI only.
- **Alternatives considered**: Only mocks (misses API changes); only live tests (slow, flaky, requires credentials); pytest-httpx (less flexible than respx for complex scenarios)

## R10: Logging Architecture

- **Decision**: Python stdlib `logging` with structured formatters and per-component loggers
- **Rationale**: Zero additional dependencies. Named loggers (`google_sdk.auth`, `google_sdk.transport`, `google_sdk.services.drive`) allow granular control. JSON formatter available for production. Credential redaction built into transport logger.
- **Alternatives considered**: structlog (additional dependency); loguru (opinionated, harder to integrate with existing logging setups); custom logging (reinventing the wheel)

## R11: Google Calendar API Specifics

- **Decision**: Use Calendar API v3 via REST
- **Rationale**: Stable, well-documented API. Key endpoints: calendars, events, freebusy. Recurring events use RFC 5545 RRULE format. Timezone handling via `dateutil` or stdlib `zoneinfo`.
- **Alternatives considered**: N/A - only one version available

## R12: Google Meet API Specifics

- **Decision**: Use Google Meet REST API v2
- **Rationale**: Provides meeting space creation, participant management, and recording access. Relatively new API (GA 2024) with straightforward REST endpoints.
- **Alternatives considered**: Using Calendar API's conferenceData (limited functionality, only creates meetings attached to events)

## R13: Rate Limiting

- **Decision**: Per-service token bucket rate limiter at transport level
- **Rationale**: Google APIs have per-service quotas. Rate limiter tracks usage and delays requests proactively rather than relying solely on 429 retries. Observable via logging.
- **Alternatives considered**: No proactive limiting (wasteful 429 retries); global rate limiter (different services have different quotas)

## R14: Project Packaging

- **Decision**: Single `google-sdk` package with optional extras (`google-sdk[keyring]`)
- **Rationale**: Simplest for users. Keyring is the only heavy optional dependency. All three services ship in one package for v1 (separate packages over-engineering at this stage).
- **Alternatives considered**: Separate packages per service (complex dependency management for v1); monorepo with multiple packages (premature)
